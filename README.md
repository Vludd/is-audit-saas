# IS Audit

**IS Audit** - локальный инструмент для анализа логов ИБ и генерации понятных отчетов с оценкой рисков, корреляцией событий и практическими рекомендациями.

Проект ориентирован на сценарий, где специалист по ИБ вручную выгружает логи из FortiGate, Bitdefender GravityZone или других источников, загружает их в систему и получает готовый отчет для технической команды и руководства.

> Цель проекта - не заменить SIEM, а быстро превращать шумные выгрузки логов в полезный аналитический документ.

---

## Зачем нужен проект

Обычные логи отвечают на вопрос:

> Что произошло?

Хороший отчет по анализу логов должен отвечать на другие вопросы:

> Насколько это опасно?
> Что нужно исправить?
> Какие проблемы требуют внимания прямо сейчас?

LogReport AI решает эту задачу за счет нормализации событий, группировки паттернов, простой корреляции между источниками и генерации отчета с помощью ИИ.

---

## Основная идея

Система построена вокруг plugin-based подхода к парсингу.

Каждый источник логов имеет собственный формат, но после парсинга все события приводятся к единой схеме.

```text
Источник логов > Парсер / Нормализатор > Единая схема событий > Корреляция / Аналитика > ИИ-анализ > Готовый отчет
```

ИИ и генератор отчетов не знают, откуда пришли данные. Они работают только с нормализованными событиями.

Это позволяет добавлять новые источники логов без переписывания всей системы.

---

## Поддерживаемые источники в MVP

Планируемые источники для первой версии:

* FortiGate - CSV/syslog export
* Bitdefender GravityZone - экспорт отчетов
* Windows Event Log - позже, через EVTX/XML export

Добавление нового источника должно сводиться к созданию одного нового парсера в директории `parsers/`.

---

## Единая схема события

Все входные данные приводятся к Common Event Format:

```json
{
  "timestamp": "2026-06-16T10:23:00Z",
  "source_type": "fortigate",
  "source_device": "FG-70G-BD",
  "category": "threat",
  "severity": "high",
  "event_type": "intrusion_attempt",
  "src_ip": "1.2.3.4",
  "dst_ip": "192.168.2.10",
  "user": null,
  "description": "Intrusion attempt was blocked by firewall policy",
  "raw": "original raw log line or source row"
}
```

Минимальная задача парсера - принять исходный файл и вернуть список событий в этой структуре.

---

## Что делает ИИ

ИИ не анализирует каждую строку лога отдельно. Это дорого, медленно и плохо масштабируется.

В MVP ИИ получает уже подготовленную агрегированную сводку:

* распределение событий по severity;
* топ источников проблем;
* подозрительные IP, хосты, пользователи;
* временные аномалии;
* сгруппированные инциденты;
* результаты корреляции между источниками.

На основе этого ИИ формирует выводы человеческим языком:

* что произошло;
* почему это важно;
* какой риск для организации;
* что нужно сделать;
* какой приоритет у проблемы.

---

## Структура итогового отчета

Генерируемый отчет должен быть полезен как техническим специалистам, так и руководству.

Планируемая структура:

1. **Исполнительное резюме**

   * общая оценка состояния;
   * топ-3 критичных проблемы;
   * выводы без технической перегрузки.

2. **Статистика периода**

   * количество событий;
   * распределение по типам и severity;
   * динамика по времени;
   * аномалии относительно нормы.

3. **Разбор инцидентов**

   * сгруппированные паттерны;
   * описание проблемы;
   * возможная причина;
   * оценка риска;
   * рекомендация.

4. **Уязвимости и нарушения политик**

   * затронутые хосты;
   * пользователи;
   * сервисы;
   * привязка к внутренним политикам ИБ.

5. **План устранения**

   * проблема;
   * действие;
   * ответственный;
   * срок;
   * приоритет.

## Примеры корреляции

### Корреляция по IP

FortiGate зафиксировал и заблокировал подозрительное соединение с хоста `192.168.2.15`.

В тот же период Bitdefender GravityZone обнаружил подозрительный процесс на этой же машине.

Вывод:

> Это не просто случайный сетевой скан. Хост может быть скомпрометирован и требует проверки.

---

### Корреляция по времени

Несколько источников одновременно фиксируют всплеск событий около `03:00`.

Вывод:

> Аномальная активность в нерабочее время требует отдельного анализа.

---

### Закрытие слепых зон

FortiGate видит IP-адрес, но не знает пользователя.

AD или Windows Event Log могут связать IP с конкретной учетной записью.

Вывод:

> Совмещение источников позволяет получить более полную картину инцидента.

## Технологический стек

### Backend

* **FastAPI** - API и загрузка файлов
* **Pydantic** - валидация Common Event Format
* **PostgreSQL** - хранение событий, анализов и отчетов
* **Redis** - задел под фоновые задачи
* **Celery** - не обязателен для MVP, добавляется при больших объемах логов

### Парсинг и обработка данных

* **polars** или **pandas** - обработка CSV
* **lxml** - XML-форматы
* **python-evtx** - Windows Event Log, вне первой версии MVP

### ИИ-слой

* **LiteLLM** - единый интерфейс к LLM-провайдерам
* Поддержка облачных и локальных моделей через конфигурацию

### Генерация отчетов

* **Jinja2** - шаблоны отчетов
* **python-docx** - DOCX-отчеты
* **WeasyPrint** - PDF-отчеты через HTML

### UI

Для MVP возможен простой интерфейс на **Streamlit**:

```text
Загрузить файл > выбрать источник > запустить анализ > скачать отчет
```

## Режим работы MVP

MVP строится как offline/local-first инструмент.

Пользователь:

1. Экспортирует логи из своей системы.
2. Запускает LogReport AI локально.
3. Загружает файлы через UI или API.
4. Получает отчет.
5. Данные не покидают локальный контур, кроме случаев, когда явно настроен внешний LLM-провайдер.

Такой подход удобен для закрытых инфраструктур, где прямой доступ к сетевому оборудованию, SIEM или EDR невозможен или нежелателен.

---

## Что не входит в MVP

В первую версию намеренно не входят:

* агентский сбор логов по API;
* real-time monitoring;
* полноценная SIEM-корреляция;
* пользовательская аутентификация;
* роли и права доступа;
* сложный rule engine;
* семантический поиск через pgvector;
* автоматическая интеграция с FortiGate, AD или GravityZone.

Эти возможности могут быть добавлены позже, но для MVP они только усложнят разработку.

## Безопасность

Логи могут содержать чувствительные данные:

* внутренние IP-адреса;
* имена пользователей;
* имена хостов;
* доменные имена;
* события антивируса и EDR;
* сведения о сетевой инфраструктуре;
* потенциальные индикаторы компрометации.

Поэтому проект должен придерживаться следующих принципов:

* не отправлять raw-логи во внешние сервисы без явной настройки;
* хранить секреты только в `.env`;
* не коммитить реальные логи в репозиторий;
* маскировать чувствительные данные в тестовых примерах;
* отделять тестовые фикстуры от реальных выгрузок;
* по возможности передавать в LLM агрегированные данные, а не сырые события.

## Принцип добавления нового парсера

Каждый новый источник должен реализовывать общий интерфейс:

```python
from typing import Iterable
from models.event import NormalizedEvent


class BaseParser:
    source_type: str

    def parse(self, file_path: str) -> Iterable[NormalizedEvent]:
        raise NotImplementedError
```

Пример:

```text
parsers/
├── fortigate.py
├── bitdefender.py
└── custom_source.py
```

После добавления парсера остальные части системы не должны меняться:

```text
Parser > Normalizer > Correlator > AI Analyzer > Report Generator
```

## Кратко

LogReport AI - это local-first инструмент для ИБ-аналитиков, который помогает превращать выгрузки логов в понятные отчеты:

```text
Логи > Нормализация > Корреляция > ИИ-анализ > Отчет
```

Главный фокус MVP - не real-time мониторинг, а быстрый offline-анализ экспортированных логов и генерация документа с выводами, рисками и планом устранения.
